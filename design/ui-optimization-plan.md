## 二、JSON Editor 重构 (需求 1+2)

### 2.1 核心数据模型

```
原始 JSON 文本 (sourceText)
    ↓ parse
配置对象 (config: NetworkConfig)  ← 单一数据源
    ↓ serialize
编辑器文本 (editorText)           ← 实时同步
    ↓ parse
可视化节点 (nodes/edges)          ← 按需重新解析
```

关键原则：
- `config` 是唯一数据源，所有修改都作用于 config
- 编辑器修改 → 解析为 config → 触发可视化刷新
- 可视化/属性面板修改 → 更新 config → 编辑器文本同步
- 下载时序列化完整 config（包含未可视化的字段如安全组）
- 永远不修改本地文件

### 2.2 节点选中 → 编辑器联动

实现方案：
1. 每个可视化节点携带 `jsonPath` 元数据（如 `vpcs.security`, `tgw.tables.pre`）
2. 用户点击节点 → 触发 `onNodeClick` → 获取 `jsonPath`
3. 根据 jsonPath 在 JSON 文本中计算行号范围
4. 编辑器滚动到该行并高亮对应区块

jsonPath 映射规则：
- Region 节点 → `{regionId}` 或根级别
- VPC 节点 → `vpcs.{vpcName}` 或 `{region}.vpcs.{vpcName}`
- TGW 节点 → `tgw` 或 `{region}.tgw`
- 路由表 → `tgw.tables.{tableName}`
- AZ 节点 → 不联动（太细粒度）
- 子网节点 → `vpcs.{vpcName}.subnets.{subnetName}`

### 2.3 编辑器功能

- 语法高亮（JSON）
- 行号显示
- 错误标记（解析失败时标红）
- 搜索/替换
- 折叠/展开 JSON 块
- "应用" 按钮：解析编辑器文本 → 更新 config → 刷新可视化 + 运行校验
- 宽度可拖拽调整

技术选型建议：使用 Monaco Editor（VS Code 的编辑器内核）或 CodeMirror 6
- 优势：原生支持 JSON 语法高亮、行号、折叠、搜索、滚动到行、范围高亮
- 如果不想引入重依赖，可以用 CodeMirror 6（更轻量）

## 三、视图模式重构 (需求 3)

### 3.1 三种视图模式

| 模式 | 说明 | 对标 |
|------|------|------|
| 架构视图 | 区域框 + VPC 框 + 子网摘要 + 路由表 + 组件标记 + 连线 | 你提供的参考架构图 |
| 详细视图 | 当前的嵌套容器布局（Region→VPC→AZ→Subnet） | 保留，优化布局 |
| 拓扑视图 | 当前的扁平节点图 | 保留，优化布局 |

### 3.2 新增"架构视图"设计

这是最重要的新增模式，目标是还原你提供的参考架构图风格：

```
┌─ 主区域 ──────────────────────────────────────────┐
│                                                    │
│  ┌─ Hub VPC (10.0.0.0/20) ──────────────────────┐ │
│  │ AZ-A: intra /28, private /24, public /24     │ │
│  │ AZ-B: intra /28, private /24, public /24     │ │
│  │ [IGW] [NAT] [NFW]                            │ │
│  └──────────────────────────────────────────────┘ │
│                    │                               │
│  ┌─ TGW (ASN 64513, 10.0.0.0/16) ────────────┐  │
│  │ ┌─ spoke ─────┐  ┌─ firewall ───────────┐  │  │
│  │ │ dev,prod→hub │  │ hub→propagate dev..  │  │  │
│  │ │ *→hub        │  │ us-east-1→peer       │  │  │
│  │ └─────────────┘  └──────────────────────┘  │  │
│  └────────────────────────────────────────────┘  │
│                    │                               │
│  ┌─ Dev VPC ─────┐  ┌─ Prod VPC ────────────┐   │
│  │ intra + private│  │ intra + private       │   │
│  └───────────────┘  └──────────────────────┘   │
└────────────────────────────────────────────────────┘
```

特点：
- VPC 用紧凑卡片表示，子网信息汇总为一行（不展开每个 AZ 的每个子网）
- 路由表内嵌在 TGW 节点中（当前详细视图已有）
- 组件（IGW/NAT/NFW/GWLB）作为 VPC 内的标签
- 区域用边框包裹
- VPC 网格排列（Hub 在上，辐射 VPC 在下方网格）
- 连线避免穿过节点

### 3.3 筛选器设计

Toolbar 中添加筛选器下拉：

```
[筛选] ▼
├─ 区域: [全部] [主区域] [us-east-1] [us-west-1]
├─ 组件: ☑ VPC  ☑ TGW  ☑ 路由表  ☐ 子网详情  ☐ Accounts
├─ 连线: ☑ TGW连接  ☑ Peering  ☐ DX
└─ 标记: ☑ CIDR  ☑ 组件标签  ☐ IP数量
```

筛选器作用于当前视图模式，控制显示/隐藏特定元素。

### 3.4 布局优化

当前问题：
1. VPCS_PER_ROW=1 导致大量 VPC 时纵向过长
2. TGW 连线从右侧出发穿过 VPC
3. 对等区域排列不够紧凑

优化方案：
1. 动态 VPCS_PER_ROW：根据 VPC 数量自动调整
   - ≤3 个 VPC: 1 列
   - 4-6 个: 2 列
   - 7+: 3 列
2. TGW 位置：从 VPC 右侧改为 VPC 下方居中（减少横向连线穿越）
3. 连线路由：使用 `pathfinding` 算法避免穿过节点
4. 对等区域：根据数量自动调整列数

## 四、属性面板重构 (需求 4)

### 4.1 面板结构

右侧面板分为两个 Tab：

```
┌─ [资源树] [添加组件] ─────────────────┐
│                                        │
│  Tab 1: 资源树                         │
│  ├─ 主区域                             │
│  │  ├─ VPC: hub (10.0.0.0/20) [HUB]  │
│  │  │  ├─ subnets: intra, public...   │
│  │  │  ├─ igw: enabled                │
│  │  │  └─ nat: enabled                │
│  │  ├─ VPC: dev (10.0.32.0/20)       │
│  │  └─ TGW (ASN 64513)               │
│  │     ├─ table: spoke                │
│  │     └─ table: firewall             │
│  └─ us-east-1                         │
│     ├─ VPC: ...                       │
│     └─ TGW (peer)                     │
│                                        │
│  点击资源 → 下方显示属性编辑表单        │
│  ──────────────────────────────        │
│  [选中: VPC hub]                       │
│  CIDR: [10.0.0.0/20]                  │
│  is_hub: [✓]                          │
│  IGW: [✓]  NAT: [✓]  NFW: [✗]       │
│  accounts: [添加]                      │
│  subnets: [编辑子网...]               │
│                                        │
│  Tab 2: 添加组件                       │
│  ├─ [+ 添加区域]                      │
│  ├─ [+ 添加 VPC]                      │
│  ├─ [+ 添加 TGW]                      │
│  ├─ [+ 添加路由表]                    │
│  ├─ [+ 添加 DX]                       │
│  └─ [+ 添加 Resolver]                │
│                                        │
└────────────────────────────────────────┘
```

### 4.2 可添加的资源类型

| 资源 | 表单字段 | 校验规则 |
|------|---------|---------|
| 区域 | 区域代码 | 格式: xx-xxxx-N |
| VPC | 名称, CIDR, 区域, is_hub, is_endpoint, az_count | CIDR 格式, 名称唯一 |
| VPC 子网 | 名称, cidrs (per AZ) | 名称规范, cidrs 数量=az_count |
| VPC 组件 | IGW/NAT/NFW/GWLB 开关 + 子配置 | NAT 需要 IGW+public |
| TGW | ASN, CIDR, peer | ASN 唯一, CIDR 唯一 |
| 路由表 | 名称, associations, propagations, routes | VPC 引用存在 |
| DX | ASN, prefixes | 仅主区域 |
| Resolver | in/out VPC, rules | VPC 引用存在 |

### 4.3 编辑流程

1. 用户在资源树中选择资源 → 下方显示属性表单
2. 用户修改属性 → 实时更新 config 缓存
3. 点击"应用"按钮 → 刷新可视化 + 运行校验
4. 或者：自动应用（每次修改后 debounce 500ms 自动刷新）

### 4.4 子网编辑器

子网是最复杂的编辑场景，需要专门的子编辑器：

```
┌─ 子网编辑器 ──────────────────────┐
│ VPC: hub (10.0.0.0/20, 2 AZ)     │
│                                    │
│ [+ 添加子网类型]                   │
│                                    │
│ intra (保留)                       │
│   AZ-A: newbits [8] netnum [0]    │
│   AZ-B: newbits [8] netnum [1]    │
│   → 10.0.0.0/28, 10.0.0.16/28   │
│                                    │
│ public (保留)                      │
│   AZ-A: newbits [4] netnum [1]    │
│   AZ-B: newbits [4] netnum [2]    │
│   → 10.0.1.0/24, 10.0.2.0/24    │
│                                    │
│ private (保留)                     │
│   AZ-A: newbits [4] netnum [3]    │
│   AZ-B: newbits [4] netnum [4]    │
│   → 10.0.3.0/24, 10.0.4.0/24    │
│                                    │
│ [应用] [取消]                      │
└────────────────────────────────────┘
```

特点：
- 实时计算并预览 CIDR
- 保留名称标记（intra/public/private）
- 自动填充 AZ 数量
- CIDR 重叠检测

## 五、实施优先级

### Phase 1: 数据模型 + JSON Editor 重构
1. 统一数据源模型（config 缓存 + 原始 JSON 保留）
2. JSON Editor 改为左侧浮动面板
3. 节点 jsonPath 映射
4. 选中联动（点击节点 → 编辑器跳转高亮）
5. 编辑器修改 → 应用到 config → 刷新可视化

### Phase 2: 布局优化
1. 动态 VPCS_PER_ROW
2. TGW 位置优化
3. 连线路由优化（避免穿越节点）
4. 对等区域布局优化

### Phase 3: 架构视图 + 筛选器
1. 新增架构视图解析器
2. 筛选器 UI + 逻辑
3. 视图切换优化

### Phase 4: 属性面板重构
1. 资源树组件
2. 属性编辑表单（VPC/TGW/路由表）
3. 子网编辑器
4. 添加组件表单（全资源类型）
5. 校验集成

## 六、技术选型建议

| 需求 | 建议 | 理由 |
|------|------|------|
| JSON 编辑器 | @monaco-editor/react | 原生 JSON 支持, 行号, 折叠, 高亮, 搜索 |
| 连线路由 | @xyflow/react 的 pathfinding edge | 避免穿越节点 |
| 表单 | 原生 React state | 项目已有模式，无需引入表单库 |
| 状态管理 | React Context 或 zustand | config 需要跨组件共享 |
